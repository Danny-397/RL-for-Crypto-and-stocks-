THE FRAMEWORK

Built from the algorithm up.

PPO, both environments, the data pipeline, and the evaluation are implemented here — no stable-baselines, no wrappers. Each layer is a separate, individually tested module.

PPO, from scratch

Clipped surrogate objective, GAE advantages, entropy bonus, orthogonal init and gradient clipping — implemented in PyTorch, not wrapped from a library.

Dual market environments

StockTradingEnv and CryptoTradingEnv share one BaseTradingEnv, so accounting, costs, and rewards never silently diverge.

Leakage control

Chronological train / validation / test splits with scalers fit on training data only. The agent is scored on data it has never seen.

Risk-aware rewards

Return net of transaction costs, plus drawdown and turnover penalties — discouraging reckless, over-leveraged noise-trading.

Honest backtests

Total return, annualised Sharpe, and max drawdown on a held-out split — the metrics a quant actually cares about.

Swap the policy or the market

The same PPO agent trades one asset or allocates across a whole basket (cross-sectional long/short). A recurrent (LSTM) policy is fully implemented, and new markets/algorithms drop in behind the same API.

ABOUT

Why I built this.

I wanted to learn modern reinforcement learning by building it, not importing it. So I implemented PPO from scratch in PyTorch, wrote two custom market environments, engineered the whole data pipeline, and set out to answer an honest question: can one agent learn to trade two very different markets?

The hardest — and most valuable — part wasn't the algorithm; it was the evaluation. An early run made the crypto agent look like it crushed the market. Instead of shipping that, I built a multi-seed significance test — and it showed the "win" was mostly luck of the random seed, statistically tied with buy-&-hold. Learning to distrust a good-looking backtest and to prove it with statistics taught me more than any single training run.

What it demonstrates: I can take a non-trivial ML system end to end — algorithm, data, training, evaluation, and a live deployment — and hold my own results to a real scientific standard.

HOW IT WORKS

Trading as a reinforcement-learning problem.

There's no labelled "right answer" for what to do on a given day — only the consequences of what you did. That's the exact setting reinforcement learning is built for: an agent takes actions in an environment, collects a reward, and learns a policy that maximises reward over the long run. Here the environment is a market, the action is a position, and the reward is risk-adjusted profit.

THE LOOP

Observe, act, get rewarded — millions of times.

Every simulated trading day, the agent runs the same three-step loop. Repeat it over millions of steps and a strategy emerges from the consequences alone — no one ever tells it the "correct" trade.

STATE

What it sees

A rolling window of 28 features — multi-horizon momentum, RSI, MACD, Bollinger %B, ATR and a volatility-regime flag, plus cross-asset signals (its strength relative to the SPY/BTC index) — concatenated with its own account state: position, cash, and normalised equity.

ACTION

What it does

One continuous number in [−1, +1] — the target position. +1 is fully long, −1 fully short, 0 is flat. The environment moves the book toward that target and charges transaction costs and slippage on every change.

REWARD

What it optimises

Not raw profit — a risk-adjusted signal (differential Sharpe) net of costs, with drawdown and turnover penalties. That's what keeps it from chasing returns by over-leveraging or churning the account into the ground.

THE ALGORITHM

What PPO actually does.

Proximal Policy Optimization is the learning rule — implemented here from the math up, no stable-baselines.

Try, then nudge

It gathers a batch of experience under the current policy, estimates which actions did better than expected — the advantage, via GAE — and shifts the policy toward those.

But not too far

A clipped surrogate objective caps how much the policy can move in one update — the "proximal" part. It's what makes PPO stable instead of collapsing, the classic failure of naive policy gradients.

Stay curious

An entropy bonus keeps it exploring early instead of locking onto one strategy too soon; orthogonal initialisation and gradient clipping keep the optimisation well-behaved.

Two heads, one network

An actor picks the position; a critic estimates how good the current state is. The critic's value is the baseline the advantage is measured against.

END TO END

From raw prices to a trader you can trust.

01

Ingest & engineer

Load real OHLCV (or generate synthetic data) and build the 28 features — returns, RSI, MACD, MA ratios, volatility, and cross-asset context.

02

Split without leakage

Chronological train / validation / test splits, with scalers fit on the training slice only — so the agent is scored purely on data it has never seen.

03

Train with PPO

Run the loop above over millions of steps, with domain randomization across tickers so the policy can't memorise one asset's path.

04

Evaluate honestly

Backtest deterministically on the held-out split, then repeat across 5 seeds and run a permutation test to see whether the edge is real.

OVERFITTING, CAUGHT

Why train across many tickers, not one.

Train the agent on a single asset and it posts absurd returns on the data it trained on — then loses money on data it has never seen. It memorised one price path. Training across many tickers (domain randomization) forces it to learn something transferable: in-sample and held-out returns line up.

Training setup In-sample return Held-out return
Stocks · single asset+14,600%−39%
Stocks · across tickers+34%+35%
Crypto · single asset+1,956,000%−57%
Crypto · across tickers+267%+250%

Those astronomical in-sample figures aren't a win — they're the overfitting tell: a single-asset agent memorises the exact training path, so it "earns" impossible returns there and then collapses out of sample. The honest column is held-out, and only domain randomization keeps it positive. (Mean across seeds; reproduced by tools/ablation.py.)

THE HONEST PART

And then: does it actually work?

A single training run can look brilliant by luck. So every result is held to a multi-seed permutation test — the whole thing is retrained under 5 independent random seeds, and the edge only counts if it survives. Watch it play out per market on the Stocks and Crypto tabs, or read the full methodology in RESULTS.md.
